# Covista AI: BigQuery Data Contract
### Version 7  |  Effective: 2026-03-30

---

## 1. Architecture Alignment

BigQuery is the source of truth only. All operational consumption flows through the following pipeline:

    Salesforce / SIS --> BigQuery --> Pub/Sub --> Firestore --> Angular App <--> Vertex AI

### Boundary of Responsibilities

**Data Engineering Team**
- Deliver structured data into BigQuery tables defined in this contract
- No Firestore logic, no UI transformations, no AI interactions
- No boolean evaluations -- pass raw values from source systems

**Application Team**
- Ingest data via Pub/Sub into Firestore
- Compute readiness risk, engagement risk, and accreditation flags in app layer
- Handle all UI rendering and on-demand Vertex AI interactions

## 3. Table Definitions

---

### 3.1 `r2c_student_profile`

One row per student per term. Core identity and programme enrolment data.

| Field | Type | Nullable | Source | Description |
|---|---|---|---|---|
| `student_id` | STRING | NO | Banner | Unique student identifier |
| `student_name` | STRING | YES | Banner | Student full name |
| `institution` | STRING | NO | Banner | Institution name |
| `program` | STRING | NO | Banner | Programme code |
| `program_name` | STRING | NO | Banner | Programme full name |
| `term_code` | STRING | NO | Banner | Academic term code (e.g. 202510) |
| `term_desc` | STRING | YES | Banner | Human-readable term description |
| `status_stage` | STRING | NO | Banner | Current student stage (Reserved, Enrolled, Dropped) |
| `enrollment_specialist_name` | STRING | YES | Salesforce | Assigned Enrollment Specialist |
| `program_start_date` | DATE | NO | Banner | Official programme start date |
| `reserve_date` | DATE | NO | Banner | Date student entered Reserve status |
| `census_date` | DATE | NO | Banner / Derived | Day 10 census milestone date |
| `funding_type` | STRING | YES | Salesforce | FAFSA or Alternative |
| `time_to_program_start_days` | INTEGER | YES | Derived | Days until programme start from snapshot date |
| `time_since_reserve_days` | INTEGER | YES | Derived | Days since Reserve date at snapshot time |
| `last_updated_at` | TIMESTAMP | NO | Pipeline | Pipeline insertion timestamp |

**Business Rules:**
- `census_date` = `program_start_date + 10 days` where not explicitly provided by Banner
- Only students with `status_stage = Reserved` at snapshot time are in scope
- `program_start_date` must be a future date at initial dataset generation

### 3.2 `r2c_student_activity_log` -- Unified Event Ledger

One row per event per student. Replaces separate task history, cases, contingency, and engagement tables. Append-only -- DE never deletes rows.

| Field | Type | Nullable | Source | Description |
|---|---|---|---|---|
| `log_id` | STRING | NO | Pipeline | UUID -- required for Pub/Sub deduplication |
| `student_id` | STRING | NO | Banner | Unique student identifier |
| `term_code` | STRING | NO | Banner | Academic term code |
| `activity_category` | STRING | NO | Various | `student_event` / `advisor_task` / `contingency` / `system_event` |
| `activity_name` | STRING | NO | Various | Standardised name -- see Section 4 |
| `activity_datetime` | TIMESTAMP | NO | Various | Timestamp of the event |
| `original_state` | STRING | YES | Various | State before event (contingencies / cases) |
| `new_state` | STRING | YES | Various | State after event |
| `communication_type` | STRING | YES | Salesforce | advisor_task rows only: `phone_call` / `email` / `text` / `chat` / `file_review` |
| `actor` | STRING | YES | Various | Who performed the action (student ID or ES name) |
| `source_system` | STRING | NO | Various | See Source System Reference below |
| `creation_date` | DATE | NO | Various | Date record was created in source system |
| `course_identification` | STRING | YES | Canvas / Banner | Course ID if applicable |
| `course_level` | STRING | YES | Banner | App uses this to evaluate `isAccredited` flag |
| `other_info` | STRING | YES | Various | Additional context (e.g. TRF on file, pending documents) |

**Business Rules:**
- `log_id` must be a UUID generated by DE pipeline -- required for idempotent Pub/Sub delivery
- `communication_type` must only be populated on `advisor_task` rows; NULL on all others
- Out-of-scope communication types must not be loaded: Auto Dialer, Social, WhatsApp, Financial Plan
- `course_level` must be passed as raw value -- accreditation evaluation is handled by app layer
- Contingency rows must be generic -- no hardcoding of transcript, nursing, or institution-specific field names
- Drop reason code `DD` (student-initiated drop) must be surfaced via `other_info` so app can adjust CTA logic

## 4. Activity Dictionary

All `activity_name` values must use the exact strings below. The app uses these to drive checklist completion and risk calculations automatically.

### student_event

| `activity_name` | `source_system` | Notes |
|---|---|---|
| `initial_portal_login` | Student Portal | Student logged in at least once |
| `fafsa_submission` | Salesforce | Exclude if `alternate_funding` selected |
| `alternate_funding` | Salesforce | Student selected non-FAFSA funding |
| `first_course_registration` | Banner | Registered to at least one accredited course |
| `wwow_login` | Canvas | Logged into WWOW orientation |
| `logged_into_course` | Canvas | Logged into at least one accredited course |
| `discussion_board_submission` | Canvas | Submitted discussion post in accredited course |
| `engagement_activity` | Various | Any qualifying activity -- definition TBC (see GAP-003) |

### advisor_task

| `activity_name` | `communication_type` | `source_system` |
|---|---|---|
| `advisor_outreach` | phone_call / email / text / chat / file_review | Salesforce |

### contingency

| `activity_name` | `source_system` | Notes |
|---|---|---|
| `contingency` | Salesforce (IAR) | One row per contingency event. Use `other_info` for institution name. Only where CONT. checkbox = TRUE |

### system_event

| `activity_name` | `source_system` |
|---|---|
| `reserved` | Banner |
| `wwow_access_granted` | Canvas |

## 6. Checklist Item Reference (for App Team)

The app team uses the following mapping to determine which `activity_name` events check off which checklist items. Included here so DE understands the downstream impact of each event.

| Checklist Item | Driven by `activity_name` | Source System | Checked Off Condition |
|---|---|---|---|
| `initial_portal_login` | `initial_portal_login` | Student Portal | Student has logged in at least once |
| `fafsa_submission` | `fafsa_submission` | Salesforce | FAFSA submitted; excluded if `alternate_funding` present |
| `course_registration` | `first_course_registration` | Banner | Registered to at least one accredited course |
| `wwow_login` | `wwow_login` | Canvas | Logged into WWOW at least once |
| `contingency` | `contingency` | Salesforce (IAR) | CONT. checkbox selected in IAR; one row per open contingency |
| `logged_into_course` | `logged_into_course` | Canvas | Logged into at least one accredited course |
| `class_participation_day10` | `discussion_board_submission` | Canvas | Discussion board post submitted in accredited course |

**Readiness Risk Thresholds (computed by App layer):**

| Checklist Item | Happy Path | Low Risk | Medium Risk | High Risk |
|---|---|---|---|---|
| `initial_portal_login` | Logged in within 24hrs of Reserve | Not logged in, 3 days since Reserve | Not logged in, 5 days since Reserve | Not logged in, 7+ days since Reserve |
| `fafsa_submission` | 4+ weeks before start | 3-4 weeks before start | 2-3 weeks before start | < 2 weeks before start |
| `course_registration` | Auto-registered | Not registered, 7-30 days before start | Not registered, 4-7 days before start | Not registered < 4 days before start |
| `wwow_login` | Logged in upon access | Not logged in within 1 week of access | Not logged in 4+ weeks after access | N/A |
| `contingency` | Cleared before start | Waiting 2 weeks - 1 month before start | Waiting < 2 weeks; TRF on file | Waiting < 2 weeks; no TRF; 4-5+ pending |
| `logged_into_course` | Logged in Day 1 | Not logged in 1+ day after start | Not logged in 2+ days after start | Not logged in 3+ days after start |
| `class_participation_day10` | Participated Day 1 | Not participated 1+ day after start | Not participated 2+ days after start | Not participated 3+ days after start |

**Engagement Risk Thresholds (computed by App layer):**

| Weeks to Programme Start | Low Risk | Medium Risk | High Risk |
|---|---|---|---|
| < 4 weeks | Activity within last 3 days | No activity > 3 days | No activity > 7 days |
| 4+ weeks | Activity within last 5 days | No activity > 5 days | No activity > 10 days |

## 7. Data Quality Rules

| Rule ID | Table | Field | Rule | Severity |
|---|---|---|---|---|
| DQ-001 | `r2c_student_profile` | `program_start_date` | Must be a future date at initial dataset generation | CRITICAL |
| DQ-002 | `r2c_student_profile` | `status_stage` | Must be `Reserved` at point of inclusion | CRITICAL |
| DQ-003 | `r2c_student_activity_log` | `log_id` | Must be unique UUID per row; duplicates rejected | CRITICAL |
| DQ-004 | `r2c_student_activity_log` | `activity_name` | Must only contain values from Section 4 dictionary | HIGH |
| DQ-005 | `r2c_student_activity_log` | `activity_category` | Must be one of: student_event, advisor_task, contingency, system_event | HIGH |
| DQ-006 | `r2c_student_activity_log` | `communication_type` | Only populated on advisor_task rows; must be NULL on all others | HIGH |
| DQ-007 | `r2c_student_activity_log` | `communication_type` | Out-of-scope types (Auto Dialer, Social, WhatsApp) must be filtered at extraction | HIGH |
| DQ-008 | `r2c_student_activity_log` | `activity_datetime` | Must not be null; flag records with null timestamps | MEDIUM |
| DQ-009 | `r2c_student_activity_log` | `contingency` rows | Must only include IAR records where CONT. checkbox = TRUE | CRITICAL |
| DQ-010 | `r2c_student_profile` | `census_date` | If not sourced from Banner, must equal program_start_date + 10 days | HIGH |
| DQ-011 | `r2c_student_activity_log` | `course_level` | Non-accredited course events must be excluded from checklist-driving activity | CRITICAL |

## 8. Known Data Gaps

| Gap ID | Area | Description |
|---|---|---|
| GAP-001 | FAFSA | 1-2 day processing delay via PowerFAIDS means submission status may lag in Salesforce. App logic must handle tolerance window. |
| GAP-002 | Course Registration | Drop reason code `DD` logic requires Banner to surface drop reason code in extract. Confirm field availability. |
| GAP-003 | Engagement Activity | Definition of `engagement_activity` not fully specified. Requires agreement on what system events qualify. |
| GAP-004 | Late Reserve Students | Students waiting for classroom access need a flag to prevent false-positive High engagement risk. Logic TBC. |
| GAP-005 | WWOW Access | `wwow_access_granted` event must be available to calculate engagement gap correctly. Confirm Canvas provides this. |
| GAP-006 | Census Date | Confirm whether census_date is sourced from Banner or always derived as program_start_date + 10. |
| GAP-007 | Contingency Institution | Institution name per contingency must be stored in `other_info`. Confirm IAR field availability. |
